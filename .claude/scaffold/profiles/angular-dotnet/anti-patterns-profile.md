# Anti-Patterns — Angular + .NET Profile

Tech-stack-specific failure modes for Angular/.NET projects. These are merged with the base anti-patterns during scaffolding.

## Frontend — Layout & Structure

- Save/submit buttons go in page header actions slot ONLY — never inside the form body
- ALL form inputs (`<input>`, `<textarea>`, `<p-dropdown>`, `<p-multiselect>`, etc.) must be inside the `<form>` element — never outside it
- Use `FormGridComponent` + `FormFieldComponent` for form layouts — never create custom grid wrappers
- Do NOT create wrapper components that duplicate `PageLayoutComponent`, `FormGridComponent`, or `PageHeaderComponent` functionality
- Every page/editor component must handle loading, empty, and error states — check the reference page for how

## Frontend — Generated Code

- NEVER edit files under `__FE_DIR__/libs/core/api/src/lib/generated/` — these are auto-generated by OpenAPI sync
- If generated types don't match what the frontend needs, fix the backend API contract and regenerate — never cast with `as unknown as` or `as any`

## Frontend — Signal & Store Patterns

- Before implementing a new page/component, read the closest existing page and follow its structure exactly
- Do NOT create new architectural patterns (new base classes, new store features, new layout systems) unless the masterplan explicitly calls for it
- Do NOT use `async/await` or `firstValueFrom` in signalStore methods — use `rxMethod` with pipe
- Do NOT use `async` pipe in templates — use `toSignal()` in the component
- Do NOT inject services directly into components — use stores for API calls and state
- Do NOT create components without `ChangeDetectionStrategy.OnPush`

## Frontend — Component Patterns

- Components must be standalone with `OnPush` change detection
- Use signal-based API (`input()`, `output()`, `viewChild()`) — no decorators (`@Input`, `@Output`, `@ViewChild`)
- Forms must extend `BaseFormComponent<T>` with `save = output<T>()` — no store injection in form components
- All user-visible text must use `| translate` pipe — no hardcoded strings
- Use `@for` with `track` — never `*ngFor`
- Lazy-load all routes with `loadComponent` — never eager `component:` in route configs

## Backend — CQRS & Handler Patterns

- All business logic goes through MediatR handlers — controllers only call `mediator.Send()` and return the result
- Do NOT bypass the audit interceptor with raw SQL INSERT/UPDATE statements
- Do NOT assign `Id = Guid.NewGuid()` or `Id = Guid.CreateVersion7()` on entities inheriting BaseEntity — EF Core handles it
- Do NOT use `throw new Exception(...)` or `throw new InvalidOperationException(...)` — use the ApiException hierarchy
- Do NOT return full DTOs from create commands — return `Guid` only (the created entity's Id)
- Use primary constructors for handlers (BE-CQRS-001)

## Backend — EF Core & Data Patterns

- Do NOT use `.ToList()` / `.ToArray()` — use collection expressions `[.. source]`
- Always use `.AsNoTracking()` in query handlers
- Use `HasConversion<string>()` for all enum properties in EF configurations
- Use `DeleteBehavior.Restrict` by default — `Cascade` requires justification comment
- Do NOT manually edit migration code files (`Migrations/2*.cs`, `AppDbContextModelSnapshot.cs`)

## Backend — State Machine Enforcement

- Status changes must go through entity domain methods — never direct assignment in handlers
- Terminal states (Completed, Expired, Revoked) must be immutable — no further transitions allowed
- Always capture old state before transitions for audit logging

## Backend — Process Management

- When the backend needs to be running (for OpenAPI regen, E2E tests), ALWAYS use health check polling — never `sleep N`
- ALWAYS stop the backend process after operations complete — never leave orphan processes
- ALWAYS build backend before starting it — never start from stale compilation
