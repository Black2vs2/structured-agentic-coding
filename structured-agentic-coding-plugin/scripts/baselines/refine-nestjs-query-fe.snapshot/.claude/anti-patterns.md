# Anti-Patterns

Known failure modes from past executions. Read by:
- **Architect** ŌĆö for task design (avoid repeating mistakes)
- **Executor** ŌĆö for prompt injection (warn dev agents)
- **Reviewer** ŌĆö for regression detection

---

## General ŌĆö Task Discipline

- Don't modify files outside task's `Files:` list ŌĆö STOP and report if other files need changes
- Don't add features beyond task description ŌĆö scope creep causes regressions
- Always run build verification after changes
- Stop and report if task seems wrong or conflicts with existing code
- Don't skip running tests
- Read existing file patterns before creating new files ŌĆö follow them exactly

### Surrendering too quickly
**Pattern:** Agent marks a task as blocked or escalates after one failed attempt.
**Rule:** Before marking anything as blocked, try at least 2 different approaches. For code issues, try a different implementation path. For test failures, try adjusting the approach. Only then escalate with: what was tried, why it failed, suggested alternatives.
**Why:** Most tasks have multiple valid solutions. The first approach failing doesn't mean the task is impossible.

### Critical thinking deficit
**Pattern:** Agent blindly follows instructions without questioning assumptions or proposing alternatives.
**Rule:** Verify assumptions before implementing. If you see a better approach than what's prescribed, propose it (but don't implement without approval). Report problems early ŌĆö don't wait until the end of a phase to mention a fundamental issue you noticed at the start. Balance critical thinking with execution speed ŌĆö don't over-analyze to the point of paralysis.

### Skipping tests silently
**Pattern:** Agent reports "tests pass" but actually skipped failing or difficult tests.
**Rule:** SKIP = FAIL. Any test report with skipped tests is treated as a failure. Tests must either run and pass, or explicitly fail with a reason. Never skip a test to make a green report.

## Frontend ŌĆö Patterns

- Read the closest existing page/component and follow its structure exactly
- No new architectural patterns (base classes, store features, layout systems) without explicit masterplan approval
- Don't modify generated/auto-generated files ŌĆö fix the source instead

## Backend ŌĆö Patterns

- All business logic through the designated handler/service pattern ŌĆö not in controllers
- Don't bypass audit or validation interceptors with raw SQL or direct DB access
- Don't return full DTOs from create operations ŌĆö return identifiers only

## Process Management

- Always use health check polling when starting services (never `sleep N`)
- Always stop backend services when done (even on failure)
- Always build before starting services

### RACE-001: Concurrent file writes
**Pattern:** Two parallel agents write to the same file simultaneously, causing merge conflicts or data loss.
**Rule:** No two agents dispatched in the same batch may have overlapping files in their `Files:` lists. The executor's dependency analysis already checks this ŌĆö if overlap exists, tasks must run sequentially, never in parallel. If dispatching parallel worktree agents, verify file disjointness before dispatch.
**Why:** Git merge conflicts from concurrent edits waste more time than sequential execution would have.

# Anti-Patterns ŌĆö Refine + nestjs-query + shadcn + Firebase + Bun

Known failure modes specific to this stack.

---

## Importing Zod from `'zod'` instead of `'zod/v4'`

**Trigger:** `import { z } from 'zod';` or `import { ... } from 'zod';` anywhere in `src/`.

**Why it's wrong:** Project uses Zod v4, which ships as a separate import path (`'zod/v4'`). Importing from `'zod'` resolves to v3, which has a different error map shape, different API for refinements, and breaks the custom error mapping in `src/lib/form-validation.ts`. The two versions are not interchangeable.

**Fix:** `import { z } from 'zod/v4';` everywhere.

**Related rules:** FE-FORM-001

---

## `useTranslation` imported from `react-i18next` instead of `@refinedev/core`

**Trigger:** `import { useTranslation } from 'react-i18next';`

**Why it's wrong:** Refine proxies `useTranslation` through its `i18nProvider` so that resource names, action labels, and notifications are translated consistently. Using `react-i18next` directly bypasses that proxy and produces inconsistent strings (especially in list/create/edit/show page titles).

**Fix:** `import { useTranslation } from '@refinedev/core';`

**Related rules:** FE-I18N-001

---

## Hand-editing `src/graphql/types.ts` or `src/graphql/schema.types.ts`

**Trigger:** Any git diff modifying these two files without a corresponding change to a `.ts`/`.tsx` file containing a `gql` tag or to `graphql.config.ts`.

**Why it's wrong:** These files are auto-generated by `graphql-codegen` (invoked via `vite-plugin-graphql-codegen` on dev-server start / file watch, or manually via `graphql-codegen-sync`). Manual edits are overwritten on the next codegen run ŌĆö and if committed, they produce merge conflicts and schema-drift bugs.

**Fix:** Change the operation (inline `gql`) or the schema (backend), then run `bun run codegen` (or `/graphql-codegen-sync`). Revert any manual edits.

**Related rules:** FE-GQL-003, FE-PROT-001, FE-PROT-002

---

## Non-bun package manager

**Trigger:** `package-lock.json`, `yarn.lock`, or `pnpm-lock.yaml` committed; `npm install` / `yarn` / `pnpm install` in CI workflows or docs.

**Why it's wrong:** Project runtime and lockfile is bun. Mixed package managers produce divergent lockfiles, different peer-dependency resolution, and install scripts that never ran on the non-intended manager ŌĆö CI breaks silently.

**Fix:** `bun install` for installs. `bun run <script>` for scripts. Remove competing lockfiles. Update any CI step that calls `npm`/`yarn`/`pnpm` directly.

**Related rules:** FE-BUN-001, FE-BUN-002

---

## `process.env.*` access in frontend code

**Trigger:** `process.env.FOO` or `process.env.VITE_FOO` in any `src/` file.

**Why it's wrong:** Vite exposes env vars via `import.meta.env` (only those prefixed `VITE_`). `process.env` doesn't exist in the browser bundle at runtime ŌĆö it either throws or resolves to `undefined` after Vite's DefinePlugin pass.

**Fix:** `import.meta.env.VITE_FOO`. Type it in `src/vite-env.d.ts` via `interface ImportMetaEnv`.

**Related rules:** FE-ENV-001

---

## Emotion `styled-components` as primary styling

**Trigger:** Widespread use of `import styled from '@emotion/styled'` in component files when a shadcn/Tailwind equivalent is available.

**Why it's wrong:** Project's primary styling is Tailwind 4 + shadcn/ui + CSS variables. Emotion duplicates style scopes, breaks theme-variable consumption, and makes dark-mode / brand-color changes require two edits instead of one.

**Fix:** Use shadcn components from `src/components/ui/`, compose with Tailwind utility classes, reference CSS variables from `src/index.css` for colors. Reserve Emotion for rare cases where Tailwind can't express the style (rare ŌĆö usually animations, document with a comment).

**Related rules:** FE-UI-004

---

## Direct `fetch()` calls bypassing Refine's dataProvider

**Trigger:** `fetch(...)` or `axios.get/post(...)` directly in a resource page / hook, calling a backend endpoint.

**Why it's wrong:** Refine's `dataProvider` (via `@refinedev/nestjs-query`) handles auth token injection, error normalization, cache invalidation via TanStack Query, and integration with `useList`/`useOne`/`useCreate` hooks. Bypassing it means no auth header, no automatic re-fetch, no optimistic updates, no access-control checks.

**Fix:** Use Refine hooks (`useList`, `useOne`, `useCreate`, `useCustom` for non-standard queries). Extend the dataProvider if a new protocol is needed.

**Related rules:** FE-REFINE-002, FE-AUTH-003

---

## Hardcoded hex colors in components

**Trigger:** `className="bg-[#132642]"`, `style={{ color: '#0D8FFD' }}`, or similar literal hex codes in component source.

**Why it's wrong:** Brand colors are declared as CSS variables in `src/index.css` (e.g., `--primary`, `--accent`, `--destructive`). Hardcoding hex breaks theme consistency, prevents dark-mode switching, and forces a code change every time design tokens evolve.

**Fix:** Reference the CSS variable: `className="bg-primary"`, `style={{ color: 'var(--accent)' }}`. Add new tokens to `src/index.css` before using them.

**Related rules:** FE-UI-003

---

## Routes declared outside the `<Refine resources>` list

**Trigger:** Routes defined via React Router's `<Route>` that do not correspond to an entry in `<Refine resources={...}>`.

**Why it's wrong:** Refine's access-control provider, breadcrumbs, navigation, and i18n all rely on the resources list. Routes outside that list silently miss those features ŌĆö a user might land on a page that bypasses role checks.

**Fix:** Register the resource in `App.tsx`'s `resources` prop. For truly non-resource routes (e.g., a marketing page), mark them explicitly and wrap with a `<Route>` under a distinct layout so the intent is clear.

**Related rules:** FE-REFINE-001

---

## Editing `patches/` without updating `package.json`'s `patchedDependencies`

**Trigger:** Any diff touching files in `patches/` without a corresponding change to `package.json`'s `patchedDependencies` block (or `trustedDependencies` / bun's patch mechanism equivalent).

**Why it's wrong:** Patches applied to dependencies (typically to Refine packages) are tracked by the package manager. Editing a patch without registering it means bun won't apply it during `bun install`, or will apply a stale version.

**Fix:** Regenerate the patch via `bun patch <package>` and `bun patch --commit <package>` (or the equivalent workflow). Ensure `package.json` references the new patch file path.

**Related rules:** FE-PROT-004
