# Scan Playbook: Internationalization

Category: `i18n` | Rules: FE-I18N-001 through FE-I18N-005

---

## FE-I18N-001 -- Zero hardcoded user-visible text

**What to check:** Flag text content in templates not using translate pipe. Flag string arguments to MessageService/ConfirmationService/DynamicDialogConfig not using translate.

**Scan 1 -- Multi-word text content in templates:**

```
Grep pattern: ">\s*[A-Z][a-z]+(\s+[a-z]+)+\s*<"
     path:    frontend/libs
     output_mode: content
     glob:    "*.html"
```

- **True positive:** `<h2>Candidate List</h2>` — hardcoded text, should use `<h2>{{ 'candidates.list.title' | translate }}</h2>`
- **False positive:** `>{{ 'key' | translate }}<` — uses translate pipe, correct
- **False positive:** `<code>some code example</code>` — code examples may not need translation
- **Confirm:** Read the template to verify the text is user-visible content (violation) vs code/technical content (acceptable).
- **Severity:** warning

**Scan 2 -- Hardcoded strings in service calls:**

```
Grep pattern: "(summary|detail|header|message)\s*:\s*['\"][A-Z]"
     path:    frontend/libs
     output_mode: content
     glob:    "*.ts"
```

- **True positive:** `this.messageService.add({ summary: 'Success', detail: 'Candidate saved' })` — should use translate
- **False positive:** `this.messageService.add({ summary: this.translate.instant('common.success'), detail: this.translate.instant('candidates.saved') })` — uses translate service
- **Confirm:** Check if the string value is a hardcoded user-visible string vs a translation key or variable.
- **Severity:** warning

## FE-I18N-002 -- Translation keys scoped to feature

**What to check:** Key prefix must match the library path. `candidates.*` keys should only appear in the `candidates/` library. `common.*` keys can appear anywhere.

**Scan:**

```
Grep pattern: "'\w+\.\w+[^']*'\s*\|\s*translate"
     path:    frontend/libs
     output_mode: content
     glob:    "*.html"
```

- **Interpretation:** Extract the key prefix (first segment before `.`) and check it matches the library folder name. Cross-feature key usage (e.g., `candidates.title` used in `invites/` library) is a violation.
- **True positive:** `{{ 'candidates.name' | translate }}` in a file under `frontend/libs/features/invites/` — cross-feature key
- **False positive:** `{{ 'common.save' | translate }}` — `common.*` keys are shared, acceptable anywhere
- **Confirm:** Check the file path to determine which library the template belongs to, then verify the key prefix matches.
- **Priority:** Medium — requires cross-referencing keys with file paths.
- **Severity:** info

## FE-I18N-003 -- Every key in both IT and EN JSON

**What to check:** Collect translation keys from code and verify both `public/i18n/it.json` and `en.json` have them.

**Scan:**

```
Grep pattern: "'[\w.]+'\\s*\\|\\s*translate|translate\\.instant\\('[\w.]+'\\)"
     path:    frontend/libs
     output_mode: content
     glob:    "*.{ts,html}"
```

- **Interpretation:** Extract unique translation keys. Then Read `frontend/public/i18n/it.json` and `frontend/public/i18n/en.json` to verify each key exists.
- **Note:** The Code Reviewer performs a complete key-to-JSON cross-reference as a Tier 2 deep check when i18n findings are detected. This scan provides initial detection.
- **Severity:** warning

## FE-I18N-004 -- No orphaned translation keys

**What to check:** Keys in JSON files not referenced in any template or TypeScript file.

**Note:** This rule is handled by the Code Reviewer as a Tier 2 deep check (complete reverse lookup: JSON keys → code references). The reviewer's Tier 1 scan pass does NOT check this — it triggers the deep check when other i18n findings are detected.

- **Severity:** info

## FE-I18N-005 -- TranslateModule in every component using translate pipe

**What to check:** If a template uses `| translate`, the component's `imports` array must include `TranslateModule`.

**Scan 1 -- Find templates using translate pipe:**

```
Grep pattern: "\| translate"
     path:    frontend/libs
     output_mode: files_with_matches
     glob:    "*.html"
```

**Scan 2 -- Check for TranslateModule in corresponding components:**

```
Grep pattern: "TranslateModule"
     path:    frontend/libs
     output_mode: files_with_matches
     glob:    "*.component.ts"
```

- **Interpretation:** For each HTML file using `| translate`, check if the corresponding `.component.ts` file imports `TranslateModule`. Match files by naming convention (e.g., `list.component.html` -> `list.component.ts`).
- **True positive:** Template uses `| translate` but corresponding component does not have `TranslateModule` in its `imports` array
- **False positive:** Template uses `| translate` and component has `TranslateModule` in imports — correct
- **Confirm:** Read the component file to verify `TranslateModule` appears in the `@Component({ imports: [...] })` array.
- **Severity:** warning
