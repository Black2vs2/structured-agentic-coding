# Scan Playbook: Internationalization

Category: `i18n` | Rules: FE-I18N-001 through FE-I18N-004

---

## FE-I18N-001 — useTranslation from @refinedev/core

**What to check:** All `useTranslation` imports come from `@refinedev/core`, never from `react-i18next`.

**Scan:**
```
Grep pattern: "import.*useTranslation.*from\\s+['\"]react-i18next['\"]"
     path:    __FE_DIR__/src
     output_mode: content
```

- **True positive:** `import { useTranslation } from 'react-i18next';` anywhere in `src/`.
- **False positive:** None — Refine proxies the hook and must be the single source.
- **Confirm:** No confirmation needed.
- **Severity:** error

---

## FE-I18N-002 — Key prefix convention

**What to check:** Translation keys used in resource pages follow `pages.<resource>.<section>.<key>`.

**Scan:**
```
Grep pattern: "t\\(\\s*['\"]"
     path:    __FE_DIR__/src/resources
     output_mode: content
```
For each match, inspect the key. Flag keys that do NOT start with `pages.`.

- **True positive:** `t('common.submit')` in a resource page (belongs under `pages.<resource>.<section>.submit`).
- **False positive:** Intentionally-shared keys under `common.*` used across pages — allowed if documented.
- **Confirm:** Read the key; if it's page-specific, it should live under pages.
- **Severity:** info

## FE-I18N-003 — No inline English strings

**What to check:** User-visible strings in JSX are wrapped in `t(...)`.

**Scan:** This rule is hard to detect cleanly. Use a heuristic:
```
Grep pattern: ">\\s*[A-Z][a-z]+\\s+[a-z]+"
     path:    __FE_DIR__/src
     output_mode: content
```
This catches JSX text nodes starting with a capitalized word followed by lowercase — likely English prose.

- **True positive:** `<Button>Submit Order</Button>` — hardcoded English.
- **False positive:** Code that happens to look like prose inside a `{t(...)}` already. Single-word labels (`<Badge>New</Badge>`) are usually fine.
- **Confirm:** Read the JSX; if there's no `t()` wrap and the text is user-facing, flag.
- **Severity:** warning

---

## FE-I18N-004 — No missing translation keys

**What to check:** Every `t('key')` call resolves to an entry in the i18n source files.

**Scan:** Requires cross-file inspection: build a set of defined keys (from i18n JSON/TS files) and a set of used keys (from `t()` calls), then diff.

**Approach:**
1. Read all files under `__FE_DIR__/public/locales/` or `__FE_DIR__/src/i18n/` (project layout may vary).
2. Extract all key paths defined.
3. Grep for `t\(['"]([^'"]+)['"]` in `src/`.
4. Diff: keys used but not defined → violations.

- **True positive:** `t('pages.orders.invalid')` used but no such key defined.
- **False positive:** Dynamic keys like `t(\`pages.${resource}.title\`)` — skip these (they can't be statically resolved).
- **Confirm:** Trace the key in the i18n source; if genuinely missing, flag.
- **Severity:** warning
